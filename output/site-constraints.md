# Ограничения из site_dir

## Прочитанные файлы
- `STUDENTS.md` — обязательный контракт симулятора: порядок работы, разрешённые визуальные и контентные изменения, защищённые элементы формы, селекторы, серверные файлы и workflow проверки конверсии.
- `AGENTS.md` (передан в контексте задачи; физического файла в `site_dir` нет) — общие инженерные соглашения: Spec-First/Contract-First, синхронизация реализации с API-контрактом, Bring Your Own Types, AAA для тестов, очистка legacy-кода при Go-рефакторинге и обязательное уточнение при выходе за рамки Swagger/OpenAPI.

Файлы `docs/**/*.md` и `.cursor/rules/**` в `site_dir` не обнаружены. Противоречий между найденными инструкциями нет.

## Можно менять
- Тексты, заголовки и описания.
- Цвета, шрифты, отступы и визуальное оформление.
- Изображения.
- Расположение элементов внутри секций.
- Видимость CTA через CSS при условии, что сами кнопки не удаляются.
- В рамках первоначальной настройки счётчика разрешено вставить код Яндекс.Метрики в `index.php` перед `</body>` на месте комментария-заглушки.
- Изменения, затрагивающие API-контракт, должны проектироваться Spec-First/Contract-First и оставаться синхронизированными с OpenAPI/Swagger/JSON Schema. Однако серверные `api/*.php` запрещены к изменению правилами симулятора, поэтому для текущего внедрения такие изменения недопустимы.
- Пользователь явно изменил контракт формы после анализа: поле `purpose` больше не требуется и удаляется. Для этой миграции разрешено синхронно изменить `index.php` и `api/submit.php`; остальные запреты сохраняются.

## Нельзя менять
- Форму `#registration-form` и её `action="api/submit.php"`.
- Поля формы `name`, `phone`, `email`.
- Скрытое поле `bot_session_id`.
- Скрипт `<script src="api/visit.php">` в `<head>`.
- Классы `.btn-register`, `.pricing-section`, `.program-module`, `.program-list`.
- PHP-файлы в папке `api/`.
- Схему `sql/schema.sql`.
- Удалять CTA-кнопки; допустимо только управлять их видимостью через CSS.
- Вносить спорные изменения, выходящие за документированные ограничения или API-спецификацию, без отдельного уточнения.

## Какие файлы/папки редактировать
- Инструкции не задают исчерпывающий список редактируемых клиентских файлов.
- `index.php` можно редактировать для разрешённых текстовых, визуальных и layout-изменений, не затрагивая защищённые форму, поля, классы и подключение `api/visit.php`.
- Можно редактировать клиентские стили и изображения, если это не удаляет CTA и не меняет защищённые селекторы/контракт симулятора.
- `api/` и `sql/schema.sql` не редактировать, кроме явно разрешённого в этой задаче удаления обязательности `purpose` из `api/submit.php`.

## Workflow после правок (из site_dir)
1. В control-panel нажать **Redeploy**; при этом данные в БД будут сброшены.
2. Снова запустить **Run visits simulator** (по умолчанию 100 визитов).
3. Проанализировать новый прогон в Яндекс.Метрике: Вебвизор, карта кликов и конверсии.
4. Сравнить конверсию до и после изменений. Учитывать, что конверсия симулятора в консоли может отличаться от Метрики.
5. При необходимости проверить локальные данные по `https://ваш-домен/api/export.php` и заявки через `https://ваш-домен/admin/orders.php`.
6. Для опционального локального запуска использовать:
   ```bash
   cd web-site
   php -S 127.0.0.1:8765
   curl http://127.0.0.1:8765/api/init.php
   ```
